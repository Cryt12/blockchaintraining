# RREPS Database

Plain SQL migrations for the Supabase PostgreSQL project. Run them in order (001 → 012) —
each file assumes the previous ones already exist because of foreign keys and shared
helpers (the `set_updated_at()` trigger function, the `generate_receipt_number()` function).

## Running the migrations

**Option A — Supabase SQL editor**
Paste each file's contents into the SQL editor and run them in numeric order.

**Option B — Supabase CLI / psql**
```bash
for f in database/migrations/*.sql; do
  psql "$SUPABASE_DB_URL" -f "$f"
done
```

`SUPABASE_DB_URL` is the direct Postgres connection string from
Supabase → Project Settings → Database (not the REST API URL).

## Schema overview

| Table | Purpose |
|---|---|
| `users` | Accounts + role (`administrator`, `property_custodian`, `verifier`, `viewer`) |
| `offices` | Organizational offices |
| `categories` | Expendable item categories |
| `employees` | Employees who return property, linked to an office |
| `expendable_items` | Property catalog with running `quantity` / `remaining_quantity` |
| `return_receipts` | One row per receipt; carries the `canonical_json` + `receipt_hash` that were sent to the blockchain |
| `receipt_items` | Line items per receipt (item, quantity, condition, remarks) |
| `blockchain_logs` | On-chain transaction metadata per receipt (tx hash, block, gas, wallet) |
| `activity_logs` | Audit trail of user actions |
| `receipt_number_counters` | Backing counter for `generate_receipt_number()` (`RRP-YYYY-NNNNNN`) |

## Design notes

- **Soft deletes** (`deleted_at`) are on `users`, `offices`, `categories`, `employees`,
  `expendable_items`, and `return_receipts` — the entities the spec calls out for
  Archive/Delete. `receipt_items`, `blockchain_logs`, and `activity_logs` are
  append-only detail/audit records, so they don't need their own soft-delete flag;
  they cascade only if a parent receipt is ever hard-deleted.
- **Receipt numbers** are generated by `generate_receipt_number()`, set as the
  `receipt_number` column default. It's backed by a per-year counter table with an
  atomic `ON CONFLICT ... DO UPDATE`, so numbering stays sequential and gapless
  under concurrent inserts and resets automatically each calendar year.
- **Blockchain data stays minimal by design**: `blockchain_logs` never stores
  receipt contents — only the hash, transaction metadata, and wallet address —
  matching the "never store personal data or full receipts on-chain" requirement.
- The Laravel backend (next phase) will connect to this same Supabase Postgres
  instance via its `pgsql` driver and model these tables with Eloquent — it does
  not need to re-run its own conflicting migrations against this schema.
